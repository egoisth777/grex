# feat-m8-release — v1.0.0 release umbrella (cargo-dist + crates.io + mdBook + pack-template + CHANGELOG/SemVer)

**Status**: draft
**Milestone**: M8 (see [`../../../milestone.md`](../../../milestone.md) §M8)
**Depends on**: M7 fully shipped to `main` 2026-04-23 (PRs #25 MCP server, #26 test harness, #28 MCP CI conformance, #29 doctor, #30 MIT/Apache-2.0 licence, #31 `grex import`); M6 scheduler + per-pack lock; M5 plugin system.

## Motivation

M7 closed the agent-facing surface (MCP) and the last three "independent" items (`import`, `doctor`, licence). The product is feature-complete for v1 per [`openspec/feat-grex/spec.md`](../../feat-grex/spec.md). What's missing is the release machinery: a way for users to actually install `grex`, a discoverable pack template, hosted docs, and a published versioning contract.

`milestone.md` §M8 enumerates five deliverables — cargo-dist binaries, crates.io publish, docs.rs + mdBook site, `grex-pack-template` reference repo, CHANGELOG + SemVer policy. All five are independent enough to land in separate PRs but cohesive enough to share one umbrella spec. None of them change runtime behaviour; all of them are distribution + discoverability work.

This change does **not** re-open any M7 surface. The four M7 residual tech-debt issues (#32, #33, #34, #35) are explicitly parked for v1.0.1 — see §Out-of-scope.

## Goal

1. **M8-1 cargo-dist wiring**: `[workspace.metadata.dist]` block, `.github/workflows/release.yml` driven by cargo-dist, matrix covers Windows / Linux / macOS × `x86_64` + `aarch64` (6 cells). Tagged `v1.0.0` push produces signed archives + installer scripts attached to a GitHub Release.
2. **M8-2 crates.io publish**: verify `grex` name availability (fallback TBD if squatted), bump all 4 workspace crates to `1.0.0` via `[workspace.package].version` inheritance, dry-run each publish, publish in dependency order (lib crates first, bin last).
3. **M8-3 mdBook docs site**: build from `.omne/cfg/*.md` sources, GitHub Pages host via `.github/workflows/docs.yml`, `[package.metadata.docs.rs]` block on each lib crate so `docs.rs` renders canonical API docs.
4. **M8-4 `grex-pack-template` reference repo**: separate repo providing a canonical pack skeleton (`.grex/pack.yaml`, standard hooks, README), linked from main README + docs, installable as a demo via `grex add <template-url>`.
5. **M8-5 CHANGELOG + SemVer policy**: `CHANGELOG.md` at repo root (Keep-a-Changelog format), `docs/semver.md` documenting what MAJOR / MINOR / PATCH mean for grex (manifest schema, CLI surface, MCP tool surface, `pack.yaml` schema).

## Design

### Stage M8-1 — cargo-dist wiring

**Tool**: `cargo-dist` (axolotl-sh/cargo-dist) — the de-facto Rust binary-release framework; used by `uv`, `ruff`, `oxc`, `oxigraph`. Generates a GitHub Actions workflow that builds, signs, archives, and publishes platform binaries to a GitHub Release on tag push.

**Config**: root `Cargo.toml` gains:

```toml
[workspace.metadata.dist]
cargo-dist-version = "0.22"            # pin the installer generator
ci = ["github"]
installers = ["shell", "powershell"]
targets = [
    "x86_64-unknown-linux-gnu",
    "aarch64-unknown-linux-gnu",
    "x86_64-apple-darwin",
    "aarch64-apple-darwin",
    "x86_64-pc-windows-msvc",
    "aarch64-pc-windows-msvc",
]
pr-run-mode = "plan"                   # plan-only on PRs; full build on tag
```

**Workflow**: `.github/workflows/release.yml` is generated by `cargo dist init && cargo dist generate`; do NOT hand-edit. Triggers on tag `v[0-9]+.[0-9]+.[0-9]+` push.

**Matrix**: 6 cells (3 OS × 2 arches). Linux `aarch64` uses cross-compilation via cargo-dist's default (docker-based `cross`); macOS `aarch64` is native on GitHub's `macos-14` runners; Windows `aarch64` uses `windows-latest` with the `aarch64-pc-windows-msvc` target (native cross-compile supported since VS 2022).

**Artifacts**: per-platform `.tar.xz` (Unix) / `.zip` (Windows) containing the `grex` binary + `README.md` + `LICENSE-MIT` + `LICENSE-APACHE`. Plus `sh` and `ps1` installer scripts that hit the release URL.

**Signing**: cargo-dist's default sigstore/cosign signing for Linux + macOS; Windows unsigned in v1.0.0 (Authenticode certificate acquisition is out-of-scope — documented follow-up).

### Stage M8-2 — crates.io publish

**Name audit**: check `https://crates.io/crates/grex`. If the name is taken or reserved:
1. Document the squatter state in the PR description.
2. Fallback name TBD — candidates (in preference order) `grex-cli`, `grex-rs`, `graphex`. Decision deferred to the audit moment, not locked in this spec.
3. If `grex` is free, reserve it immediately on the first dry-run attempt.

**Version bump**: add `[workspace.package]` `version = "1.0.0"` (the licence block landed in M7-4c already exists here). Every crate's `Cargo.toml` replaces its own `version = "0.0.1"` with `version.workspace = true`. Affects 4 crates: `grex-core`, `grex-mcp`, `grex`, plus any lib split already present.

**Publish order** (lib → bin, bottom-up dep graph):
1. `grex-core` (depended on by everything else).
2. `grex-mcp` (depends on `grex-core`).
3. `grex` (bin crate; depends on both libs).

Each step uses `cargo publish --dry-run -p <crate>` before the real publish. Dry-runs block merge via CI; real publishes run manually from a release tag checkout.

**Metadata per crate**: `description`, `keywords` (max 5), `categories` (pick from crates.io's taxonomy — `command-line-utilities`, `development-tools`, `development-tools::cargo-plugins`), `readme = "README.md"`, `repository`, `homepage`, `documentation = "https://docs.rs/<crate>"`.

### Stage M8-3 — mdBook docs site

**Source**: `.omne/cfg/*.md` already holds the design docs. mdBook wraps them into a navigable site.

**Structure**: `docs/book/book.toml` points `src = "../../.omne/cfg"` (relative from book root). A `SUMMARY.md` is added under `.omne/cfg/` listing chapter order; mdBook reads it as the TOC root. `SUMMARY.md` is the only net-new file under `.omne/cfg/`; existing design docs are untouched.

**Build**: `cargo install mdbook` in CI, `mdbook build docs/book` emits static HTML into `docs/book/book/`.

**Hosting**: `.github/workflows/docs.yml` on push-to-main builds and deploys to `gh-pages` branch via `actions/deploy-pages`. Settings: Pages source = "GitHub Actions". Custom domain deferred to follow-up (use `*.github.io` default in v1).

**docs.rs**: each lib crate's `Cargo.toml` gains:

```toml
[package.metadata.docs.rs]
all-features = true
rustdoc-args = ["--cfg", "docsrs"]
```

This produces canonical API docs at `docs.rs/grex-core`, `docs.rs/grex-mcp`. The mdBook site covers narrative / design / concept docs; `docs.rs` covers API reference. The two are cross-linked but not merged.

### Stage M8-4 — `grex-pack-template` reference repo

**Repo**: a new separate GitHub repo (`grex-org/grex-pack-template`) — NOT a subdirectory of this repo. Created as part of this change's PR workflow but tracked under its own git history.

**Contents**: the minimum viable pack skeleton.
- `.grex/pack.yaml` — declarative pack-type with one illustrative `file-write` action and one `git-clone` action.
- `.grex/hooks/pre-install.sh`, `.grex/hooks/post-install.sh` — commented-out stubs showing the hook surface.
- `README.md` — "How to use this template" + `grex add <this-url>` demo instructions.
- `LICENSE` — dual MIT/Apache-2.0 (match the main repo).
- `.gitignore` — tracks the M6 managed-block contract.

**Discoverability**: main repo's `README.md` gets a "Getting Started" section linking to the template repo. mdBook site gets a dedicated chapter (`.omne/cfg/pack-template.md` — new file) walking through the template structure.

**Demo**: the `grex` main README "quick start" becomes: `grex init && grex add https://github.com/grex-org/grex-pack-template && grex run <pack-name>`. A screenshot / asciinema recording of this flow is attached to the release notes (nice-to-have, not blocking).

### Stage M8-5 — CHANGELOG + SemVer policy

**CHANGELOG.md**: new file at repo root, [Keep-a-Changelog 1.1.0](https://keepachangelog.com/en/1.1.0/) format. Initial `1.0.0` entry rolls up M1-M7 as "Added" bullets referencing their merged PRs (no per-commit noise). `[Unreleased]` section at the top is seeded empty — post-1.0.0 PRs are expected to append their own entries.

**docs/semver.md**: new file defining what MAJOR / MINOR / PATCH mean for grex. Draft shape:

- **MAJOR**: breaking changes to any of (a) `grex.jsonl` manifest schema, (b) `pack.yaml` declarative schema, (c) CLI verb name or argument shape, (d) MCP tool name or parameter schema.
- **MINOR**: new verbs, new MCP tools, new pack-types, new manifest row kinds that older readers skip cleanly, new CLI flags with safe defaults.
- **PATCH**: bug fixes, performance improvements, doc corrections, test additions. No new surface.
- **Deprecation policy**: a deprecated verb / tool / flag warns for one MINOR cycle before removal (which is MAJOR). `doctor` surfaces deprecation warnings in its output.
- **The manifest wire**: `grex.jsonl` row `schema_version` field (already on every row post-M3) is the hard invariant — writers never emit older versions; readers MUST handle unknown future fields as "skip, don't error".

### Cross-stage integration points

- Tag `v1.0.0` push triggers M8-1 release.yml → artefacts land on GitHub Release → README quick-start's install script URL is `sh` / `ps1` from that release.
- Crates.io metadata's `documentation` URL points at `docs.rs` (M8-3); `homepage` points at the mdBook GitHub Pages URL.
- `grex-pack-template` README links to both the crates.io page and the mdBook site.
- CHANGELOG is referenced by the GitHub Release body (auto-extracted by cargo-dist from `CHANGELOG.md`'s `[1.0.0]` section).

## Known risks

1. **crates.io name squatting**: `grex` is a short four-letter name. If squatted / reserved, the fallback adds friction (rename across docs + cargo-dist artefact names + installer scripts). Mitigation: audit on day one of M8-2; if squatted, pick fallback before starting other stages so artefact names are consistent.
2. **cargo-dist arm64 cross-compile flakes**: Linux `aarch64` via docker `cross` has historically been slower and occasionally OOMs on GitHub-hosted runners. Mitigation: if flaky, bump the runner to `ubuntu-latest-4-cores` (free tier has this) or drop to a 5-cell matrix (defer Linux arm64 to v1.0.1) rather than hold the release.
3. **mdBook hosting path**: `actions/deploy-pages` requires Pages to be enabled in the repo settings and the default branch to be `main`. Both currently hold; document the dependency in `docs.yml` header. A custom domain would add DNS coordination — explicitly deferred.
4. **Windows Authenticode signing absent**: `cargo-dist` produces unsigned `.exe`s for Windows in v1.0.0. Users will see SmartScreen warnings. Documented in release notes + `docs/semver.md` (not a SemVer issue, but worth calling out). Follow-up: acquire a code-signing cert for v1.1.0.
5. **grex-pack-template lives in a separate repo**: creation + bootstrapping happens during M8-4 work but is not tracked in this repo's git history. Coordination via a checklist in the M8-4 PR; cross-link the template repo's first commit SHA in the `[1.0.0]` CHANGELOG entry for traceability.
6. **`workspace.package.version` collision with M7-4c licence block**: the `[workspace.package]` block was introduced in M7-4c for licence inheritance. M8-2 adds `version` to the same block; verify the merge is clean and no crate has a stale `version = "0.0.1"` after the refactor.

## Out-of-scope (v1.0.1 parking lot)

The four M7 residual tech-debt issues are **parked for v1.0.1** and explicitly NOT blockers for v1.0.0:

- **#32** — [M7 doctor: TOCTOU handling in on-disk drift check — race between `symlink_metadata` and report emission].
- **#33** — [M7 MCP: `-32002` code overload disambiguation (split into distinct pack-op vs init-state codes)].
- **#34** — [M7 doctor: `--fix` severity roll-up edge case when post-fix retry still surfaces warnings].
- **#35** — [M7 MCP: pre-init request gate + double-init gate missing (rmcp 1.5.0 limitation documented in feat-m7-1 spec §Known limitations)].

Each is tracked in its own GitHub issue, assigned to `v1.0.1` milestone. Rationale: none is a correctness bug for the v1.0.0 happy path; all are edge-case hardening that can safely wait one PATCH cycle. Shipping v1.0.0 on schedule matters more than absorbing these into the release umbrella.

Other non-goals:

- **No `grex upgrade` / self-update mechanism** — deferred to v2 per `milestone.md` v2 backlog.
- **No pack registry (`grex.dev`)** — deferred to v2.
- **No custom domain for docs site** — default `*.github.io` only in v1.0.0.
- **No Windows code-signing** — follow-up for v1.0.x when a cert is acquired.
- **No `grex-inst` exemplar pack listing** — the milestone mentions it in passing; downgraded to a README link, not a separate stage.
- **No CHANGELOG automation (release-please, towncrier)** — hand-maintained in v1; automation is a v1.1.0+ concern.

## Effort estimate

`milestone.md` §M8 budgets **2-4 days**. Split across the 5 stages:

- M8-1 cargo-dist wiring — ~1 day (config + first successful matrix run; arm64 cross-compile is the wildcard).
- M8-2 crates.io publish — ~0.5 day (version bump + dry-runs + actual publish; contingent on name audit).
- M8-3 mdBook + docs.rs — ~0.5 day (mdBook is boilerplate; docs.rs metadata is 6 lines × 2 crates).
- M8-4 grex-pack-template — ~1 day (separate repo setup + README + demo verification on 3 OS).
- M8-5 CHANGELOG + SemVer — ~0.5 day (prose work; no code).

Total: **~3.5 days** of focused work, fitting the 2-4 day band with margin for arm64 / name-audit surprises.
